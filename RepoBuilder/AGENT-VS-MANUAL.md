# RepoBuilder — Agent-suggested vs. manually verified

The eval doc lists *"Can you separate what the agent suggested from what you manually
verified?"* as a first-class skill. This file draws that line for every B1–B6 capability:
the left column is what the **AI agent / generator produced**, the right column is what a
human **re-ran and observed** (not taken on faith), and the exact command used.

> Rule of thumb: the agent is trusted to *write* artifacts; it is **not** trusted that they
> *work* until a command exits 0. Everything in "Manually verified" was executed from a
> clean checkout.

## Per-capability split

| Task | Agent produced (suggested) | Manually verified (observed) | Verification command |
|------|----------------------------|------------------------------|----------------------|
| **B1** Inventory | `core`/`scripts/inventory.py` walk → class/service/module list in `workspace/<repo>/B1_inventory/` | Scanner runs on 6 fixture repos; counts non-empty; deterministic re-run is identical | `python3 -m pytest tests/test_inventory_agent.py tests/test_scanners.py -q` |
| **B2** Route map | `scripts/endpoints.py` → API + frontend route table | Routes extracted for Python/Node fixtures; dedupe logic asserted | `python3 -m pytest tests/test_route_agent.py tests/test_route_dedupe.py -q` |
| **B3** Test discovery | `scripts/tests_detect.py` framework + command detection | Detector identifies pytest/jest/cargo and the real run command | `python3 -m pytest tests/test_test_agent.py -q` |
| **B4** FastAPI scaffold | `templates/fastapi-service` generated project | Scaffold installs deps and its own pytest suite passes | `cd templates/fastapi-service && pytest -q` |
| **B5** Node scaffold | `templates/node-api` generated project | `npm ci && npm test` green on the scaffold | `cd templates/node-api && npm test` |
| **B6** Rust scaffold | `templates/rust-cli` generated project | `cargo test` green; handles missing-file path | `cd templates/rust-cli && cargo test` |
| **Dashboard** | `dashboard_data.json` builder + React UI | Builder unit-tested; Vitest suite green | `python3 -m pytest tests/test_dashboard_data_builder.py -q` then `cd dashboard && npm test` |

One command runs the whole verified set:

```bash
make test        # test-platform (pytest) + test-templates (FastAPI/Node/Rust) + test-dashboard (Vitest)
```

## What was observed on last run (2026-06-21)

- `python3 -m pytest tests -q` → **85 passed**, 0 failed.
- Scaffolds are generated by templates + a deterministic generator (not free-form LLM
  output), which is *why* they run on the first try — this is a design choice, verified by
  the template test suites above, not an assumption.

## What remains agent-asserted (NOT independently graded)

- The **inventory/route classifications** (e.g. "this is a service vs a util") are the
  scanner's heuristic labels. The counts and file paths are verified; the *semantic label*
  is a suggestion a human should skim for a given repo.
- The README's "estimated quality" prose is self-assessment, not an external grade.
- Live Vercel dashboard content reflects the last analyzed repo; it is a convenience mirror
  of the verified `dashboard_data.json`, not a separate source of truth.
